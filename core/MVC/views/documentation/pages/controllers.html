Controllers
THE MVC framework should actually be named C[MV] (Controller, Model, View) as this is the actual process! The controller sits as the primary junction between a URL being called and the action which will take place. In essense if you create a new url scheme in the routing file (./configuration/routing.json) you can establish a new with a controlling object (The Controller).
The basics of the Controller
Unless you create a more complex routing configuration the basic execution of the controller is done in the init() method. This is a requirement of the controller object and it will be run by default for a URL. Within this you handle the preparation of the page you want to display. If you had simply wanted to display a static html file then you could simply load a view or you begin to add more complicated elements. What follows is the simplist answer to a controller. The user has requested a url and the init method has been called by default. The controller has responded by rendering out a view.
public function init() { $this -> view("documentation/index.html", array()); }
Now, the controller also acts as the middle man between the view (what the user sees) and the data (what we show the user). The controller should therefore not handle actual data (e.g. doing some SQL query) but instead offload those tasks to other libraries, in this case, models. First, here's another basic controller which builds a simple form of data to submit to the view.
public function init() { $data = array("title" => "My Webpage Title"); $this -> view("documentation/index.html", $data); }
If we want a more complicated bit of data to pass to the view then we could do the following:
public function init() { $thisModel $data = array("title" => "My Webpage Title"); $this -> view("documentation/index.html", $data); }
